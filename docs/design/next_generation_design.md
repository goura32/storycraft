# 次世代実装の設計方針

> この文書は、[製品仕様](../product/SPECIFICATION.md)を実装へ落とすための最小方針である。
> ここにない内部構造、ライブラリ、保存形式は固定しない。

## 1. 正本状態

正本は、次の五種類だけに分ける。

| 正本 | 役割 |
|---|---|
| 企画と全巻計画 | シリーズの目的、巻順、主要項目を決める |
| 台帳 | 人物、関係、世界、時間、伏線の現在状態を持つ |
| 場面成果物 | 本文、引継ぎ要約、許可済み状態更新を一組で持つ |
| 進捗 | 完了済み工程、次に実行する工程、停止理由を持つ |
| 実行記録 | 生成、検証、批評、修正、採用の根拠を残す |

場面を採用するときは、本文、引継ぎ要約、状態更新を分けて採用しない。状態更新を検証して台帳へ適用できたときだけ、場面全体を完了にする。

次の工程は、正本状態から必要な情報だけを読み取る。過去の草稿、未採用の修正版、関係のない台帳項目を正本として参照してはならない。

## 2. 工程の共通契約

LLMが生成する各工程は、同じ流れで処理する。

```text
入力正本 → 草稿 → 検証 → 批評 → 修正版 → 検証 → 採用または草稿保持 → 保存
```

- 草稿と修正版は、同じ工程の出力契約を満たす。
- 検証に失敗した候補は採用しない。
- 批評・修正の失敗は、検証済みの草稿を失わせない。
- 採用結果は、採用した候補と根拠を結び付けて保存する。
- 再開時は、保存済みの採用結果だけを使う。

工程固有の検証は追加してよい。ただし、採用の流れや正本の意味を工程ごとに変えてはならない。

## 3. 状態更新の境界

状態を変えられるのは、場面採用だけである。

1. 場面カードから、参照できる台帳項目と更新できる項目を決める。
2. 草稿と修正版は、その範囲だけを参照・更新できる。
3. 採用時に、更新対象、変更可能な現在状態、場面IDを検証する。
4. 検証済み更新を台帳へ適用する。
5. 更新済み台帳を次場面へ渡す。

固定情報、秘密の真実、回収条件、未許可項目は、いかなる場面でも変更できない。

## 4. 受け入れテスト

破壊的改修は、次を満たすまで完成扱いにしない。

| 確認 | 成功条件 |
|---|---|
| 指定巻数 | 指定した巻数と、全巻計画・出力巻数が一致する |
| 状態反映 | 採用した場面の許可済み更新だけが、次場面の台帳状態へ反映される |
| 情報境界 | 未許可の台帳参照・更新、未知ID、固定情報の変更を拒否する |
| 再開 | 中断後、企画を再入力せず、保存済みの採用状態から続行できる |
| 完結 | 初回企画由来の主要な問い、主要項目、結末の未回収があれば完成にしない |
| 出力 | 指定巻数のMarkdownと全巻Markdownに、空本文・必要な章の欠落・同一本文の重複がない |

受け入れテストは実装より先に作る。各項目は、正常例と拒否例を一つ以上持つ。

## 5. 実装順

1. 正本状態と保存・読込
2. 共通の生成・検証・批評・修正・採用の流れ
3. 場面採用時の状態更新
4. 再開と出力
5. 受け入れテスト全件

旧実装との互換処理や移行処理は作らない。次世代の正本と契約だけを実装する。
