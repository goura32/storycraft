# プロンプトテンプレートと出力契約

> この文書は、次世代実装が実際に送信するプロンプトと採用前の検証契約を対応付ける設計資料である。製品仕様は[製品仕様](../product/SPECIFICATION.md)を正本とする。

## 正本と記録

- 実送信テンプレートは `src/storycraft/nextgen_model.py` の `_STAGE_CONTRACTS` と `_CRITIQUE_CONTRACT` である。
- 採用可否は `src/storycraft/nextgen.py` の工程別検証器が決める。JSON object モードだけを信用して採用してはならない。
- 生の要求と応答はシリーズ作業場所の `raw/` に、草稿・批評・修正版・失敗理由・採用済み状態は `state.json` に残す。
- 生成、批評、修正のすべてで、入力に必要な正本状態だけを渡す。修正工程は候補、批評、元の入力を渡し、同じ工程契約を再掲する。

## 全巻計画 `plan`

| 項目 | プロンプトで説明する期待 | 採用前の検証 |
|---|---|---|
| `volumes` | 依頼された巻数と一致する4〜10巻の配列 | 4〜10件、依頼巻数と完全一致 |
| `volumes[].number` | 1から始まる欠番・重複なしの連番 | 実行順と一致する連番 |
| `volumes[].title` | その巻だけを表す空でない巻題 | 空文字列を拒否 |
| `volumes[].chapters` | 少なくとも1章の配列 | 空配列を拒否 |
| `chapters[].number` | 巻内で1から始まる欠番・重複なしの連番 | 実行順と一致する連番 |
| `chapters[].title` | 空でない章題 | 空文字列を拒否 |
| `change` | 出来事列挙でなく、その巻の読了後に成立する変化または区切り | 空文字列を拒否 |
| `leaves_question` | 最終巻以外では未解決で次巻を必要にする問い。最終巻は空文字列でよい | 最終巻以外の空文字列を拒否 |

## 場面カード `scene_card`

| 項目 | プロンプトで説明する期待 | 採用前の検証 |
|---|---|---|
| `scene_id` | 入力の実行対象IDを変更せず返す | 実行対象IDとの完全一致 |
| `visible_thread_ids` | この場面で本文上参照してよい `available_thread_ids` の最小集合 | 未知ID・重複IDを拒否 |
| `allowed_update_ids` | 実際に進展または回収する、可視IDの部分集合。最終場面では未回収IDをすべて含める | 未知ID・重複ID・不可視ID・最終場面の未許可未回収IDを拒否 |

## 場面成果物 `scene`

| 項目 | プロンプトで説明する期待 | 採用前の検証 |
|---|---|---|
| `content` | 注釈、箇条書き、Markdown、執筆メモを含まない完成日本語本文。約500〜5000字を目安にする。最終場面は初回企画の結末に到達する | 空本文を拒否 |
| `handoff_summary` | 本文で起きた事実、終点、未解決点だけを具体化した引継ぎ要約。本文外の事実を加えない。最終場面には予告や未確定の結末を残さない | 空文字列を拒否 |
| `state_updates` | `allowed_update_ids` だけを使う更新配列 | 配列でない値、未知ID、未許可ID、同一IDの重複更新を拒否 |
| `state_updates[].id` | 更新する主要な問いのID | 許可済み台帳IDに限定 |
| `state_updates[].status` | 本文の事実に対応する `open`、`in_progress`、`resolved` のいずれか | 列挙値以外を拒否 |

## 完結確認 `closure`

| 項目 | プロンプトで説明する期待 | 採用前の検証 |
|---|---|---|
| `resolved_ids` | 入力台帳の全IDを一度ずつ。追加、欠落、別名、重複は不可 | 台帳ID集合との完全一致、重複なし |
| `ending_reached` | 初回企画の結末が採用済み場面で実際に到達したときだけ `true` | `true` 以外を拒否し、全台帳項目が `resolved` であることを確認 |

## 批評 `critique`

| 項目 | プロンプトで説明する期待 | 採用前の検証 |
|---|---|---|
| `issues` | 確認できる契約違反だけを示す配列。問題がなければ空配列 | 配列でない値を拒否 |
| `issues[].severity` | 下流工程への影響で選ぶ `critical`、`major`、`minor` | 列挙値以外を拒否 |
| `issues[].field` | 問題の出力フィールド名またはJSONパス | 空文字列を拒否 |
| `issues[].description` | 入力契約、ID、状態、結末条件に照らした具体的な問題 | 空文字列を拒否 |
| `issues[].suggestion` | 修正工程が直接適用できる具体的な修正指示 | 空文字列を拒否 |

批評は称賛、点数、出版可否、曖昧な感想を出してはならない。候補にない問題を作らない。

## 修正 `revise`

修正工程の出力スキーマは、対象の生成工程と完全に同じである。プロンプトは対象工程の全スキーマ、意味、制約、相互整合を再掲する。批評で指摘されていない正しいID、順序、本文事実を失ってはならない。修正版は草稿と同じ検証器で再検証する。

## レビュー手順

1. 各工程の検証器で必須・列挙・相互整合・採用条件を抽出する。
2. 実際に保存された `raw/*_generate_*.json`、`raw/*_critique_*.json`、`raw/*_revise_*.json` の `sent_messages` に、対応する項目の意味、形式、制約、整合があることを確認する。
3. 実LLMの応答を検証器に通し、草稿、批評、修正版、採否、失敗理由が `state.json` に残ることを確認する。
4. 完走後、全巻数、章見出し、非空本文、本文重複、主要な問い、結末、Markdown出力を監査する。
