# Storycraft 仕様

> この文書は、現在の実装方針を記載する。要件を守る範囲で変更してよい。
> 変えてはいけない製品上の約束は[要件](REQUIREMENTS.md)を参照する。

## 1. 初回入力

利用者は最初に次を入力する。

- 仮題
- ジャンルと雰囲気
- 主人公
- 重要人物
- 起きてほしいこと
- 避けたいこと
- 最後にどう終えたいか
- 希望巻数。省略時は指定しない

希望巻数を指定した場合は4〜10の範囲だけを受け付ける。指定しない場合、LLMは5巻を第一候補として、企画に合う4〜10巻を選ぶ。

## 2. 巻・章・場面数の自動設計

巻数、各巻の章数、各章の場面数は、LLMが初回企画と物語上の役割から選ぶ。利用者やプログラムが「5巻なら8章、1章3場面」のように固定しない。

| 対象 | 範囲 | 選び方 |
|---|---:|---|
| シリーズの巻数 | 4〜10巻 | 5巻を第一候補に、題材と物語の広がりで選ぶ |
| 1巻の章数 | 6〜10章 | 巻の役割、転換点、必要な本文量で選ぶ |
| 1章の場面数 | 2〜4場面 | 緊張、会話、移動、転換の必要性で選ぶ |
| 1場面の本文量 | 1,800〜2,600字 | 場面の役割に合わせて選ぶ |
| 1巻の本文量 | 32,000〜70,000字 | 章数・場面数・場面量の組み合わせで満たす |

各巻は少なくとも18場面を持つ。場面数と本文量の見積りがこの範囲を外れる計画は採用しない。

章数・場面数がたまたま同じになっても、確認・作り直しは行わない。構成の妥当性はLLMの物語設計に委ねる。

## 3. 生成の流れ

全巻本文や全章の細部を一度に作らない。長い文脈を持ち続けず、必要な情報だけを次の単位へ渡す。

```text
初回企画
  ↓
全巻計画
  ↓
登場人物カードと関係カード
  ↓
世界・場所・組織・重要物の台帳
  ↓
時間・期限台帳
  ↓
伏線・重要イベント台帳
  ↓
第1巻の章一覧
  ↓
第1章の場面カード
  ↓
第1章を場面ごとに執筆
  ↓
第2章の場面カード
  ↓
……
  ↓
第1巻の要約
  ↓
第2巻の章一覧
  ↓
……
  ↓
最終巻の執筆・結末回収
  ↓
巻別Markdownと全巻Markdown
```

### 3.1 全巻計画

最初の呼び出しで、次を含む全巻計画を作る。

- 選んだ巻数
- 各巻の題、役割、主人公と重要人物の関係変化
- 各巻で解決する問題と、次巻へ残す問い
- 最終巻で回収する結末
- 各巻の章数
- シリーズ全体の視点規則。語りの形式、視点変更を許す単位、視点人物候補

全巻計画はシリーズの方向、各巻の章数、視点規則だけを決める。章内の場面や本文の細部はこの時点で決めない。

### 3.2 登場人物カードと関係カードを作る

全巻計画の直後、伏線・重要イベント台帳を作る前に、主要人物と繰り返し登場する人物のカードを作る。台帳はこの人物カードの`id`を参照する。

**人物カードの対象**

- `主要`: 主人公、相手役、主要な対立者など、シリーズの結末へ影響する人物
- `準主要`: 複数の章または巻へ登場し、人物関係や事件へ継続的に影響する人物
- `一時`: 一つの場面または章だけで役割を終える人物。人物カードは作らず、場面カードだけで扱う

同じ人物を別名で重複登録しない。呼び名、肩書、愛称は一枚の人物カードの`aliases`へ集める。

**人物カードの項目**

| 項目 | 内容 | 更新するか |
|---|---|---|
| `id` | 変更しない識別子 | しない |
| `name`、`aliases` | 本名、呼び名、肩書、愛称 | 原則しない |
| `role` | 主人公、相手役、対立者、協力者など | 原則しない |
| `narrative_function` | 物語内で果たす働き | 原則しない |
| `voice_note` | 語彙、文の長さ、敬語、言いよどみなど、会話に必要な特徴 | 原則しない |
| `behavior_note` | 迷い方、怒り方、他者への接し方など、行動に現れる特徴 | 原則しない |
| `appearance_cues` | 本文で繰り返す必要がある外見上の手掛かりだけ | 原則しない |
| `initial_goal`、`initial_constraint` | 物語開始時の目的と制約 | しない |
| `current_goal`、`current_pressure` | 現在の目的、迷い、差し迫った問題 | 場面後に必要な場合だけ更新する |
| `current_location`、`current_condition` | 現在地、負傷、拘束、離脱など、次の場面に効く状態 | 場面後に必要な場合だけ更新する |
| `current_knowledge` | 日常的な事実として本人が知ること | 場面後に必要な場合だけ更新する |
| `last_updated_at` | 状態を最後に更新した場面番号 | 更新する |

人物カードには、正体、秘密、犯行の真実、恋心の最終的な告白、回収が必要な約束を直接書かない。それらは伏線・重要イベント台帳の`core_fact`にだけ置き、人物カードは関係する台帳の`id`だけを参照する。

人物カードの例は次とする。

```json
{
  "id": "char-01",
  "name": "朝倉澪",
  "aliases": ["澪", "朝倉さん"],
  "role": "主人公",
  "narrative_function": "父の残した記録を追い、自分の選択を引き受ける中心人物",
  "voice_note": "短く言い切る。追い詰められると質問で返す。",
  "behavior_note": "助けを求める前に一人で調べようとする。恐れるほど手を動かす。",
  "appearance_cues": "雨に濡れた髪を耳へ掛け直す癖。",
  "initial_goal": "父の失踪の理由を知る。",
  "initial_constraint": "父の件を他人へ話すことを避けている。",
  "current_goal": "暗号の読み方を確かめる。",
  "current_pressure": "相手役を信じるべきか判断できない。",
  "current_location": "旧資料室",
  "current_condition": "行動可能",
  "current_knowledge": "暗号が父の筆跡で書かれていると知っている。",
  "thread_ids": ["thread-01"],
  "last_updated_at": "volume-01/chapter-02/scene-03"
}
```

**関係カード**

人物同士の現在の関係は、人物カードへ双方から複写せず、一組につき一枚の関係カードで管理する。

```json
{
  "id": "rel-01",
  "character_a_id": "char-01",
  "character_b_id": "char-02",
  "current_state": "協力しているが、互いに重要な情報を伏せている。",
  "reader_knowledge": "二人が互いを完全には信じていないことを読者は知っている。",
  "relationship_thread_id": "thread-04",
  "last_updated_at": "volume-01/chapter-02/scene-03"
}
```

恋愛、信頼、対立、和解のように最終的な変化を回収する関係は、`relationship_thread_id`で伏線・重要イベント台帳へつなぐ。関係カードは現在地、台帳は変化の導入・回収を正本とする。

**場面生成へ渡す人物情報**

本文生成では登場人物全員のカードを渡さない。対象場面に必要な人物だけを渡す。

- 視点人物: 固定プロフィール、現在状態、関係カード、許可された知識
- 視点人物以外: 固定プロフィール、外から観察できる現在状態、関係カード
- 秘密・正体・将来の関係変化: 人物カードではなく、対象場面の伏線台帳から許可された範囲だけ渡す

これにより、視点人物以外の内心、秘密、将来の心変わりを、地の文が勝手に説明しない。

**場面後の更新**

場面本文の応答には、実際に変化した人物状態と関係状態だけを含める。固定プロフィールは更新しない。

```json
{
  "character_updates": [
    {
      "character_id": "char-01",
      "field": "current_knowledge",
      "value": "相手役が暗号の形式を知っていたと知る。"
    }
  ],
  "relationship_updates": [
    {
      "relationship_id": "rel-01",
      "new_state": "協力を続けるが、互いに相手の隠し事を疑っている。"
    }
  ],
  "entity_updates": [
    {
      "entity_id": "place-01",
      "field": "current_state",
      "value": "第1巻第1章の後、職員用の鍵で入室された。"
    }
  ],
  "timeline_updates": [
    {
      "timeline_id": "time-01",
      "status": "進行中"
    }
  ]
}
```

更新対象は人物カードの`current_`で始まる項目、関係カードの`current_state`、世界台帳の`current_state`、時間・期限台帳の`status`と`actual_scene`、および各台帳の`last_updated_at`に限る。更新には必ず場面番号を記録する。固定項目、台帳の真実、回収条件を場面執筆が書き換えることはできない。

### 3.3 世界・場所・組織・重要物の台帳を作る

登場人物カードと関係カードを作った後、複数場面にまたがるか、後で使うか、状態が変わる世界要素だけを台帳へ登録する。対象は次である。

- 繰り返し登場する場所
- 組織、集団、家、店、学校など
- 所有者、所在、利用条件が物語へ影響する重要物
- 物語上の制約になる世界の規則
- 固有名、略称、専門用語

一度だけ背景に出る通行人、一般的な家具、本文で二度と使わない店名は登録しない。秘密の由来、隠された機能、後で明かす真実はこの台帳に置かず、伏線・重要イベント台帳へ置く。

| 項目 | 内容 | 更新するか |
|---|---|---|
| `id` | 変更しない識別子 | しない |
| `kind` | 場所、組織、重要物、世界の規則、用語 | 原則しない |
| `name`、`aliases` | 正式名、略称、別名 | 原則しない |
| `stable_fact` | 作者と本文で共有できる確定事実 | 原則しない |
| `use_or_access_rule` | 入れる条件、使える条件、代償、制限 | 原則しない |
| `current_state` | 所在、所有者、開放・閉鎖、稼働・破損、勢力関係など | 必要な場合だけ更新する |
| `related_character_ids` | 関係する人物 | 必要時だけ更新する |
| `thread_ids` | 秘密や回収が必要な項目の参照 | 必要時だけ追加する |
| `last_updated_at` | 状態を最後に更新した場面番号 | 更新する |

```json
{
  "id": "place-01",
  "kind": "場所",
  "name": "旧資料室",
  "aliases": ["資料室"],
  "stable_fact": "市役所の地下にあり、古い記録を保管している。",
  "use_or_access_rule": "平日夜は施錠され、職員用の鍵が必要。",
  "current_state": "第1巻開始時点では閉鎖されていない。",
  "related_character_ids": ["char-01"],
  "thread_ids": ["thread-01"],
  "last_updated_at": "volume-01/chapter-01/scene-01"
}
```

場面本文には、対象場面の場所・組織・重要物・規則だけを渡す。隠された機能や秘密は、伏線台帳の許可範囲がなければ渡さない。

### 3.4 時間・期限台帳を作る

シリーズ開始からの時間、移動、締切、回復期間、同時進行する重要な出来事を管理する。正確な日付が物語に不要な場合は、`第1巻開始から3日目の夕方`のような相対時刻を使う。

すべての会話や食事を登録しない。登録するのは、順序や経過時間を誤ると物語が壊れるものだけである。

| 項目 | 内容 |
|---|---|
| `id` | 変更しない識別子 |
| `kind` | 時点、期限、移動、回復期間、同時進行イベント |
| `description` | 何がいつ起きるか |
| `starts_at`、`ends_at` | 相対時刻または必要な場合だけ暦日 |
| `related_ids` | 人物、場所、伏線・重要イベントのID |
| `status` | 予定、進行中、完了、失効 |
| `actual_scene` | 実際に起きた場面番号 |

```json
{
  "id": "time-01",
  "kind": "期限",
  "description": "資料室の改装工事が始まり、記録へ入れなくなる。",
  "starts_at": "第1巻開始から5日目の朝",
  "ends_at": "第1巻開始から5日目の朝",
  "related_ids": ["place-01", "thread-01"],
  "status": "予定",
  "actual_scene": null
}
```

各場面カードには開始時刻、終了時刻、場所IDを持たせる。本文生成には、現在の時刻、対象場面の時刻、移動や期限に関係する台帳項目だけを渡す。場面が完了したら、実際に進んだ時刻と期限の状態を更新する。

### 3.5 伏線・重要イベント台帳を作る

登場人物カード、世界台帳、時間・期限台帳を作った後、章と場面を設計する前に、未回収にしてはいけない伏線と重要イベントを台帳へ登録する。

台帳に入れるのは、最終的な説明、発見、対決、約束の実現のように、回収が必要なものだけである。雰囲気づけの小道具、一度限りの会話、細かな演出は台帳に入れない。

**台帳の項目**

| 項目 | 内容 | 本文生成での用途 |
|---|---|---|
| `id` | 変更しない識別子 | 場面カードと実際の更新を結び付ける |
| `kind` | 謎、秘密、約束、関係、脅威、手掛かり、世界の規則など | 見せ方を選ぶ |
| `importance` | `主要`または`補助` | 完結前の確認対象を決める。主要だけが未回収で停止する |
| `core_fact` | 作者役のLLMだけが常に知る真実 | 場面で矛盾しないようにする。本文へ自動で出してはいけない |
| `involved_characters` | 関係する人物 | 人物ごとの知識を管理する |
| `reader_knowledge` | 現時点で読者が知っている範囲 | 早すぎる説明や種明かしを防ぐ |
| `character_knowledge` | 現時点で各人物が知っている範囲 | 視点人物が知らない情報の漏れを防ぐ |
| `presentation_rule` | 行動、会話、観察、内心、文書など、情報をどう見せるか | 情報を説明文で済ませない |
| `clue_plan` | 回収前に見せる手掛かりと、その見せ方 | 唐突な回収を防ぐ |
| `must_not_reveal_before` | 直接説明・正体・解決を出してはいけない時点 | 先走った種明かしを防ぐ |
| `introduce_by` | 導入する期限 | 導入忘れを防ぐ |
| `resolve_by` | 回収する期限 | 回収忘れを防ぐ |
| `resolution_condition` | 何が本文中で起きれば回収と見なすか | 完結前の確認基準にする |
| `status` | `未導入`、`進行中`、`回収済み` | 現在地を示す |
| `introduced_at`、`last_advanced_at`、`resolved_at` | 実際に操作した場面番号 | 回収の根拠にする |

各項目の例は次とする。

```json
{
  "id": "thread-01",
  "kind": "謎",
  "importance": "主要",
  "description": "主人公の父が残した暗号の意味",
  "core_fact": "暗号は父が主人公を危険から遠ざけるために残した選択記録である",
  "involved_characters": ["主人公", "相手役"],
  "reader_knowledge": "暗号の存在だけを知る。意味と父の意図は知らない。",
  "character_knowledge": "主人公:暗号を見つけたが読めない。相手役:暗号の形式を知っているが由来は知らない。",
  "presentation_rule": "暗号の意味は説明しない。文字の欠け、父の筆跡への主人公の反応、相手役のためらいで見せる。",
  "clue_plan": "第1巻で筆跡の違和感、第3巻で同じ記号を使う文書を見せる。",
  "must_not_reveal_before": "第4巻までは父の意図を本文で直接説明しない。",
  "introduce_by": "第1巻第2章",
  "resolve_by": "最終巻第6章",
  "resolution_condition": "暗号が読まれ、父の選択が主人公の最終判断へ影響する。",
  "status": "未導入",
  "introduced_at": null,
  "last_advanced_at": null,
  "resolved_at": null
}
```

台帳の項目は、導入先と回収先が決まらなければ採用しない。`主要`は導入・回収・情報開示の条件まで必須とし、`補助`は必要な場合だけ登録する。

章や場面の設計中に新しい主要伏線・重要イベントが必要になった場合は、次の章の場面カードを作る前に台帳へ追加し、回収先と情報開示の条件も決める。本文を先に書いてから、主要な伏線だけを後追いで管理することはしない。

**本文へ渡す情報の範囲**

本文生成には全台帳を渡さない。対象場面に関係する項目だけを、次の4つに分けて渡す。

1. 作者だけが知る真実。本文で直接明かしてよいとは限らない。
2. 視点人物が現在知っていること。
3. 読者が現在知っていること。
4. この場面で新しく見せてよいこと、見せてはいけないこと、見せ方。

視点人物が知らない事実は、その人物の内心、地の文、断定的な描写で書かない。秘密を知る人物が視点人物でも、`must_not_reveal_before`がある場合は、内心で真実を言い切らず、行動、沈黙、言い換え、観察できる反応で表す。

### 3.6 巻の章一覧を作る

巻を書き始める直前に、その巻の章一覧を一度だけ作る。入力は次である。

- 全巻計画の該当巻
- 初回企画の重要部分
- それまでに完成した巻の短い要約
- 台帳の未回収項目
- 最終巻なら結末条件

出力は章数ぶんの章一覧である。各章には次だけを含める。

```json
{
  "chapter_number": 1,
  "title": "章題",
  "purpose": "この章が物語内で果たす役割",
  "start_state": "章の開始時点",
  "end_state": "章の終了時点で必ず変わること",
  "scene_count": 3
}
```

`scene_count`は2〜4で、全巻計画で決めた章数ぶんだけ章を返す。本文や場面の詳細を含めないため、応答構造を浅く保てる。

最終巻の章一覧を作る前に、台帳で`回収済み`ではない項目を確認する。未回収の項目は、まだ書いていない最終巻の章へ回収先として配置する。

### 3.7 章の場面カードを作る

各章は順番に扱う。第1章を書き終えるまで第2章の場面カードは作らない。

章の場面カードを作る入力は次である。

- 対象章の章一覧
- 初回企画の必要部分
- 前章の最後の引継ぎ要約。第1章では巻の開始状態
- その巻で既に起きた重要な変化
- この章で導入・進展・回収する台帳項目
- 最終章なら、最終巻で回収すべき結末条件

出力は対象章の場面数ぶんだけの場面カードである。

```json
{
  "scene_number": 1,
  "viewpoint_character": "主人公",
  "start_time": "第1巻開始から1日目の夜",
  "end_time": "第1巻開始から1日目の夜",
  "location_id": "place-01",
  "entity_ids": ["place-01"],
  "purpose": "この場面の役割",
  "opening": "始まる状況",
  "required_events": ["必ず起きること"],
  "characters": ["登場人物"],
  "thread_actions": [
    {"id": "thread-01", "action": "導入"}
  ],
  "reader_disclosure": "読者には暗号の存在と主人公の不安だけを見せる。意味と父の意図は明かさない。",
  "presentation_rules": [
    "暗号の説明ではなく、欠けた文字と主人公の反応で見せる",
    "主人公が知らない父の意図を地の文で断定しない"
  ],
  "end_change": "場面の終わりに変わること"
}
```

`thread_actions`は、台帳項目を`導入`、`進展`、`回収`する場合だけ記載する。`viewpoint_character`、`start_time`、`end_time`、`location_id`、`reader_disclosure`、`presentation_rules`はすべての場面カードで必須とし、`viewpoint_character`は全巻計画の視点規則に従う。`location_id`と`entity_ids`は世界・場所・組織・重要物の台帳を参照する。場面カードは1章あたり2〜4件だけである。各場面をすべて書き終えた後の結果を、次章の場面カード作成へ渡す。

**情報の見せ方に工夫が必要なケース**

| ケース | 台帳で決めること | 本文での扱い |
|---|---|---|
| 謎・秘密・正体 | 読者と各人物の知識、直接明かしてはいけない時点、手掛かり | 真実を説明しない。観察、反応、欠けた情報、矛盾する行動で見せる |
| 読者だけが知る危険 | 読者が知る時点と、視点人物が知らない範囲 | 視点人物を不自然に賢くしない。読者だけが危険を理解できる配置にする |
| 秘密を持つ人物の視点 | 内心で言い切ってよい範囲、伏せる理由 | 真実を内心で説明しない。迷い、話題の回避、身体反応、選択で示す |
| 恋心・関係の変化 | 当人と読者が気付く段階、関係を変える条件 | 感情名の説明を連発せず、距離、言葉の選び方、約束、行動の優先順位で見せる |
| 後で使う道具・世界の規則 | 先に見せる時点、実際に使う場面、利用条件 | 回収時に初登場させない。事前に自然な使用、観察、会話で存在と条件を見せる |
| どんでん返し | 手掛かり、隠す核心、明かす時点 | 手掛かりは本文に置くが、結論を直接説明しない。回収時に前の描写とつなげる |
| 文書・手紙・告白 | 直接説明してよい情報、読者と人物の受け取り方 | 文書の内容をそのまま説明する必要がある場合だけ、直接開示を許可する |

`presentation_rules`に特別な指定がない場合でも、情報は行動、対立、会話、観察を通じて示す。地の文だけで設定や感情を説明し続けない。

### 3.8 場面を順に執筆する

場面カードができたら、番号順に本文を書く。

```text
第1章
  場面1の本文 → 引継ぎ要約、台帳、人物・関係状態を保存
  場面2の本文 ← 場面1の引継ぎ要約 → 保存
  場面3の本文 ← 場面2の引継ぎ要約 → 保存

第2章
  場面カードを作る ← 第1章最後の引継ぎ要約、未回収台帳、人物・関係状態
  場面1の本文 → 保存
  ……
```

LLMへ渡すのは次だけである。

- 場面カード
- 視点人物、読者への開示範囲、本文の見せ方
- 対象場面に関係する台帳項目の作者だけが知る真実
- 視点人物と読者が現在知っていること
- 対象場面の場所・組織・重要物・世界の規則
- 対象場面の開始時刻、終了時刻、関係する期限・移動
- 必要な人物情報の抜粋
- 直前場面の引継ぎ要約
- 同じ章で既に起きた重要な変化
- 対象場面の`thread_actions`
- 最終場面だけ、結末を回収する条件

LLMからは本文、引継ぎ要約、実際に起きた台帳更新を受け取る。台帳更新は、場面カードで指定した`thread_actions`のIDと操作に一致するものだけを受け付ける。`導入`または`進展`を受け付けた台帳項目は`進行中`へ、`回収`を受け付けたものは`回収済み`へ更新する。各場面の完了後に保存するため、中断しても前の場面は作り直さない。

### 3.9 巻の要約と次巻への接続

一巻を書き終えたら、各章の最後の引継ぎ要約から一巻の短い要約を作る。次巻には本文全体を渡さず、この要約、未解決の問い、人物関係の現在地、台帳の未回収項目だけを渡す。

最終巻の章一覧、最終章の場面カード、最後の場面には、初回企画の結末条件を必ず渡す。

### 3.10 完結前の確認

最終場面の執筆後、台帳の主要項目がすべて`回収済み`か確認する。

確認では、台帳、各場面の台帳更新、引継ぎ要約をLLMへ渡す。LLMは各項目について、回収した場面番号または未回収を返す。`回収済み`の台帳項目に対応する場面番号がない場合は未回収として扱う。

未回収項目がある場合、シリーズを完成扱いにせず、最終巻の未出力部分を再計画する。すでに全場面を書き終えて回収先がない場合は、未回収項目と停止理由を出力し、完結済みMarkdownを作らない。

表現上の軽い含みや余韻まで機械的に判定しない。台帳へ登録した主要項目だけを対象にする。

### 3.11 出力

全巻の場面が完成し、台帳の主要項目がすべて回収済みになった後、プログラムが本文を順に連結してMarkdownを作る。この段階でLLMへ再び本文を書かせない。

### 3.12 分割生成の基本方針

一度の呼び出しで多くの情報を生成させると、弱いLLMほど品質が落ち、指定を守らなくなる。そのため、すべての生成を小さな単位へ分割する。

- 一つの呼び出しは、種類の異なる成果物を混ぜず一種類に絞る。
- 計画は全巻、巻、章、場面の順に段階的に細分化し、巨大な一括JSONは作らない。
- 場面本文は必ず1場面ずつ書く。
- 場面カードは1章あたり2〜4件とし、一章全体を一度のJSONで返させない。
- 各呼び出しへ渡す文脈も最小限に絞る。全巻本文、全設定、全台帳を毎回渡さない。
- 渡すのは、直前場面の引継ぎ要約、対象場面に関係する台帳の抜粋、確定済みの事実、その場面の制約だけとする。

### 3.13 生成内容の改善（LLM自己改善）

検証に通った出力を、同じLLMへ渡して改善させると散文の品質が上がることが多い。ただし、構造化データの計画・カード・台帳は、改善パスで構造が壊れるリスクが高く、first-releaseでは対象外とする。改善の対象は場面本文（散文）に絞る。

改善パスと再試行は別である。再試行はJSON構造や制約の検証失敗に対するもので、同じ入力で再度生成する。改善は、検証にすでに通った本文を別の呼び出しで良くするものである。

場面本文生成の後、設定で有効なら1回だけ改善パスを実施する。

改善パスへの入力は次である。

- 草稿となった場面本文
- その場面の場面カード（視点、開示範囲、見せ方、thread_actions）
- 関連する台帳項目の抜粋と現在状態
- 改善の方向（あれば。例：引きの強さ、地の文の削り、対話の自然さ）

改善パスは生成時と同じJSON形状を返す。`content`、`handoff_summary`、`thread_updates`、`character_updates`、`relationship_updates`、`entity_updates`、`timeline_updates`を含め、検証も生成時と同一とする。

改善で次を壊してはならない。

- 既に確定した事実と台帳の状態
- 読者への開示範囲と秘密の隠し方
- 各場面の文字数目安
- 人物の話し方と一貫した行動

改善パスの応答が検証に失敗した場合は、草稿を維持して先へ進む。改善パスを再試行して無限ループしてはならない。改善は1パスを上限とし、first-releaseでは複数パスは行わない。

## 4. 台帳IDと参照の規則

### 機械採番

すべての台帳IDはプログラムが機械的に採番し、作成後は変更・再利用しない。LLMはIDを採番しない。

| 台帳 | ID形式 | 例 |
|---|---|---|
| 人物 | `char-0001` | `char-0001` |
| 人物関係 | `rel-0001` | `rel-0001` |
| 世界・場所・組織・重要物 | `entity-0001` | `entity-0001` |
| 時間・期限 | `time-0001` | `time-0001` |
| 伏線・重要イベント | `thread-0001` | `thread-0001` |

採番器はプロジェクトごとの連番を保存する。削除したID、失敗したLLM応答で出たID、過去の実行で使ったIDを再利用しない。

新規の台帳項目をLLMへ作らせる場合、LLMは内容だけを返す。プログラムが内容を検証して保存した後にIDを割り当てる。LLMが新しいIDを返しても採用しない。

### LLMへ既存項目を渡す方法

LLMに既存項目を参照、選択、更新させるとき、IDだけの候補一覧を渡してはならない。対象となる各項目について、必ずID、内容、選択に必要な現在の文脈を一緒に渡す。

```text
[thread-0007]
種類: 謎
内容: 主人公の父が残した暗号の意味
現在の読者知識: 暗号の存在だけを知る
現在の状態: 進行中
この場面で可能な操作: 進展

[thread-0012]
種類: 約束
内容: 主人公が弟へ「戻る」と約束した
現在の状態: 未導入
この場面で可能な操作: なし
```

本文生成、章の場面カード、台帳確認、状態更新のいずれでも同じ規則を使う。LLMにIDの文字列から意味を推測させない。

### 機械処理する応答の検証

LLM応答を機械的に処理するフィールドは、既知IDまたはその呼び出しで定義したenum値だけを受け付ける。自由文を受け付けるのは本文、要約、説明、状態の記述など、機械的な選択をしないフィールドだけである。

```text
既知IDの例:
  char-0001
  rel-0002
  entity-0004
  time-0003
  thread-0007

定義済みenumの例:
  thread action: 導入 / 進展 / 回収
  thread status: 未導入 / 進行中 / 回収済み
  timeline status: 予定 / 進行中 / 完了 / 失効
  update field: current_goal / current_pressure / current_location /
                current_condition / current_knowledge / current_state
```

応答を受け取ったプログラムは、構造、必須項目、型、既知ID、許可されたIDの候補集合、enum値、ID間の参照関係を検証する。

未知ID、未知enum、許可されていない既知ID、IDの形式不正、参照先の不在は、すべてスキーマ不正と同じ扱いにする。名称の近さから推測補正したり、未知IDを自動作成したり、別IDへ勝手に置き換えたりしない。

基本対応は、同じ入力文脈とID付き候補一覧を使った一回の再試行である。二度目も不正なら、その作業単位で停止し、保存済みの状態を残す。

## 5. Ollamaの利用方法

### 接続設定

OllamaのOpenAI互換APIを使う。初期設定は次とする。

```yaml
llm:
  base_url: http://ws1.local:11434/v1
  model: qwen3.6:35b-a3b-mtp-q4_K_M
  thinking: true
  stream: true
  first_event_timeout_seconds: 3600
  idle_timeout_seconds: 600
```

`base_url`と`model`は設定または環境変数で変更できる。OpenAIクライアントはAPIキーを必須とするが、Ollamaでは値を使わないため`ollama`を渡す。

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://ws1.local:11434/v1",
    api_key="ollama",
    timeout=None,
)
```

呼び出し全体の制限時間は設けない。初回受信と、受信後の無応答だけを監視する。

### Thinkingとストリーミング

すべて `POST /v1/chat/completions` を使う。Thinkingとストリーミングは常に有効にする。

```python
stream = client.chat.completions.create(
    model=settings.llm.model,
    messages=messages,
    response_format={"type": "json_object"},
    stream=True,
    extra_body={"think": True},
)
```

実測では、`qwen3.6:35b-a3b-mtp-q4_K_M`のOpenAI互換ストリームは、本文より先に`delta.reasoning`へthinkingの断片を返した。受信処理はすべてのchunkから次を集める。

- `delta.reasoning`: thinkingの断片
- `delta.content`: 最終的な本文またはJSONの断片
- 終了情報

thinkingは本文として使わない。本文は`delta.content`だけを連結して読む。

### 無応答の判定

本文がまだ届いていなくても、thinkingが届いている間は正常に実行中と扱う。

- 接続から最初のthinkingまたは本文まで: 1時間待つ
- 最初の受信後、thinking・本文・終了情報のいずれも届かない時間: 10分で停止する
- 呼び出し全体の上限: 設けない

受信したchunkごとに、時刻・種別・文字数だけを記録し、thinkingまたは本文を受け取るたびに最終受信時刻を更新する。thinking本文は保存・表示しない。画面やコマンドラインには「思考中」「本文生成中」「保存中」の進捗だけを出す。

### JSONの検証

構造化した応答が必要な依頼では、`response_format`だけに頼らない。依頼文の末尾にも、その呼び出し専用の小さなJSON Schemaを含める。

本文のstreamが終了した後、次の順に扱う。

1. `delta.content`を連結する。
2. JSONとして厳密に読み取る。
3. アプリ側のモデル定義で必須項目・型・文字列長を検証する。
4. 機械処理するIDが既知であり、この呼び出しで渡した候補一覧に含まれること、enum値がこの呼び出しで定義した値に含まれること、ID間の参照先が存在することを検証する。
5. `result`、`data`、`series`、`volume`、`scene`の入れ子に対象のJSONがあり、すべての検証に通る場合だけ取り出す。
6. 読み取りまたは検証に失敗した場合は、同じ依頼を一度だけ再試行する。
7. 二度目も失敗した場合は、その作業単位で停止し、保存済みの状態を残す。

JSON Schemaは浅く保つ。深い入れ子、大量の選択肢、複雑な正規表現は使わない。

この手順の再試行は、検証失敗への対応である。検証を通った出力をさらに良くする改善パス（§3.13）は別の呼び出しであり、ここでの再試行回数には含まない。

### LLMへの依頼単位

| 順番 | 依頼 | 応答 | 目的 |
|---:|---|---|---|
| 1 | 全巻計画 | 巻数、巻の役割、章数、結末、視点規則 | シリーズの骨格を決める |
| 2 | 登場人物・関係カード | 人物の固定プロフィール、開始時の状態、関係の現在地 | 人物の正本を作る |
| 3 | 世界・場所・組織・重要物の台帳 | 固有名、確定事実、利用条件、開始時の状態 | 世界要素の正本を作る |
| 4 | 時間・期限台帳 | 時点、締切、移動、回復期間、同時進行イベント | 時間の制約を作る |
| 5 | 伏線・重要イベント台帳 | 回収が必要な項目、導入先、回収先 | 回収対象を先に決める |
| 6 | 巻の章一覧 | 章の役割、開始・終了状態、場面数 | 一巻の順番を決める |
| 7 | 章の場面カード | 2〜4件の場面カード、視点、時刻、場所、開示規則、台帳操作 | 一章を執筆可能な単位へ分ける |
| 8 | 場面執筆 | 本文、引継ぎ要約、全台帳の実更新 | 物語本文を作る |
| 8b | 場面本文の改善（任意） | 草稿の改善版（同じJSON形状） | 散文の品質を上げる。1パス上限、失敗時は草稿維持 |
| 9 | 巻の要約 | 人物関係、未解決の問い、重要な事実 | 次巻へ必要な情報を縮める |
| 10 | 完結前の台帳確認 | 各項目の回収場面番号または未回収 | 未回収を完成扱いにしない |
| 11 | 必要時の場面修正 | 修正後の本文、引継ぎ要約 | 壊れた形式または空本文を直す |

場面執筆の応答は次の小さなJSONに限定する。

```json
{
  "content": "場面本文",
  "handoff_summary": "次の場面に渡す要約",
  "thread_updates": [
    {"id": "thread-01", "action": "導入"}
  ],
  "character_updates": [
    {
      "character_id": "char-01",
      "field": "current_knowledge",
      "value": "相手役が暗号の形式を知っていたと知る。"
    }
  ],
  "relationship_updates": [
    {
      "relationship_id": "rel-01",
      "new_state": "協力を続けるが、互いに相手の隠し事を疑っている。"
    }
  ],
  "entity_updates": [
    {
      "entity_id": "place-01",
      "field": "current_state",
      "value": "第1巻第1章の後、職員用の鍵で入室された。"
    }
  ],
  "timeline_updates": [
    {
      "timeline_id": "time-01",
      "status": "進行中"
    }
  ]
}
```

## 6. 生成失敗時

本文が空、必要な見出しが欠ける、JSONが読めない、または場面の必要項目が欠ける場合は、その場面だけ再試行する。

再試行は1回までとする。二度目も失敗した場合は、シリーズ全体を破棄せず、その場面で停止する。

表現の好み、軽い矛盾、細かな文章品質は停止条件にしない。必要なら一度だけ書き直し、それでも残る問題は記録して先へ進む。

## 7. 保存と再開

実装では、次だけを保存する。

- 初回企画
- 全巻・章・場面の計画
- 登場人物カード、関係カード、場面ごとの状態更新
- 世界・場所・組織・重要物の台帳と場面ごとの状態更新
- 時間・期限台帳と場面ごとの状態更新
- 伏線・重要イベント台帳と場面ごとの更新
- 各場面の本文と引継ぎ要約
- 各依頼の生の入力と最終本文応答
- ストリーミングの受信時刻・種別・文字数
- 完了した最後の場面
- 停止理由。停止した場合だけ

同じ設定を複数形式で複写しない。再開時は、最後に完了した場面の次から始める。

## 8. 出力

出力先の初期形は次とする。

```text
output/
  volume-01.md
  volume-02.md
  ...
  volume-10.md
  series.md
```

実際に作る巻数に対応するファイルだけを出力する。

各巻には巻題、章題、本文を含める。`series.md`には全巻を順番に収録する。販売用情報の自動作成、EPUB生成、KDPへの入稿は行わない。

## 9. KDP向けの扱い

初巻を無料導入巻として扱うが、KDPでの無料配布の実現方法は、公開時点のKDP規約と販促制度に従う。

初期版はKDPへの自動入稿を行わない。出力したMarkdownを利用者が確認・整形して公開する。
