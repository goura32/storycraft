# Storycraft 仕様

> この文書は、現在の実装方針を記載する。要件を守る範囲で変更してよい。
> 変えてはいけない製品上の約束は[要件](REQUIREMENTS.md)を参照する。

## 1. 初回入力

利用者は最初に次を入力する。

- 仮題
- ジャンルと雰囲気
- 主人公
- 重要人物
- 起きてほしいこと
- 避けたいこと
- 最後にどう終えたいか
- 希望巻数。省略時は指定しない

希望巻数を指定した場合は4〜10の範囲だけを受け付ける。指定しない場合、LLMは5巻を第一候補として、企画に合う4〜10巻を選ぶ。

## 2. 巻・章・場面数の自動設計

巻数、各巻の章数、各章の場面数は、LLMが初回企画と物語上の役割から選ぶ。利用者やプログラムが「5巻なら8章、1章3場面」のように固定しない。

### 選択できる範囲

| 対象 | 範囲 | 選び方 |
|---|---:|---|
| シリーズの巻数 | 4〜10巻 | 5巻を第一候補に、題材と物語の広がりで選ぶ |
| 1巻の章数 | 6〜10章 | 巻の役割、転換点、必要な本文量で選ぶ |
| 1章の場面数 | 2〜4場面 | 緊張、会話、移動、転換の必要性で選ぶ |
| 1場面の本文量 | 1,800〜2,600字 | 場面の役割に合わせて選ぶ |
| 1巻の本文量 | 32,000〜70,000字 | 章数・場面数・場面量の組み合わせで満たす |

各巻は少なくとも18場面を持つ。場面数と本文量の見積りがこの範囲を外れる計画は採用しない。

### 単調な構成を防ぐ条件

LLMは、全巻の構成表に巻・章・場面の数を記載する。アプリは次を機械的に確認する。

- シリーズ全体で、少なくとも2種類の章数が使われている。
- 同じ章数の巻が3巻連続しない。
- 各巻で、少なくとも2種類の場面数が使われている。
- 全シリーズで、2場面・3場面・4場面のうち少なくとも2種類が使われている。
- 1巻の場面数と本文量が範囲内に収まる。

満たさない場合は、LLMへ「物語上の役割を変えずに、章と場面の配分だけを作り直す」と一度だけ依頼する。これは品質採点ではなく、同じ形の繰り返しを避けるための構造確認である。

## 3. 生成の流れ

全巻本文や全章の細部を一度に作らない。長い文脈を持ち続けず、必要な情報だけを次の単位へ渡す。

```text
初回企画
  ↓
全巻計画
  ↓
第1巻の詳細計画
  ↓
第1巻の場面を順に執筆
  ↓
第1巻の要約
  ↓
第2巻の詳細計画
  ↓
…
  ↓
最終巻の執筆・結末回収
  ↓
巻別Markdownと全巻Markdown
```

### 3.1 全巻計画

最初の呼び出しで、次を含む全巻計画を作る。

- 選んだ巻数
- 各巻の題、役割、主人公と重要人物の関係変化
- 各巻で解決する問題と、次巻へ残す問い
- 最終巻で回収する結末
- 各巻の章数と場面数の配分

この計画はシリーズの方向を決めるが、各場面の細部までは決めない。完成した巻の内容によって、後続巻の詳細計画は調整できる。

### 3.2 巻の詳細計画

各巻を執筆する直前に、その巻だけの詳細計画を作る。入力は次である。

- 全巻計画の該当巻
- 初回企画の重要部分
- それまでに完成した巻の短い要約
- 未解決の問いと、最終巻なら結末条件

出力は章ごとの役割と、各章に含める場面カードである。場面カードには、場面の目的、必須の出来事、登場人物、次へ残す変化だけを含める。

### 3.3 場面執筆

場面は順番に作る。LLMへ渡すのは次だけである。

- 場面カード
- 必要な人物・世界設定の抜粋
- 直前場面の引継ぎ要約
- 同じ章で既に起きた重要な変化
- 最終場面だけ、結末を回収する条件

LLMからは本文と引継ぎ要約を受け取る。各場面の完了後に保存するため、中断しても前の場面は作り直さない。

### 3.4 巻の要約と次巻への接続

一巻を書き終えたら、各場面の引継ぎ要約から一巻の短い要約を作る。次巻には本文全体を渡さず、この要約、未解決の問い、人物関係の現在地だけを渡す。

最終巻の詳細計画と最後の場面には、初回企画の結末条件を必ず渡す。

### 3.5 出力

全巻の場面が完成した後、プログラムが本文を順に連結してMarkdownを作る。この段階でLLMへ再び本文を書かせない。

## 4. Ollamaの利用方法

### 接続設定

OllamaのOpenAI互換APIを使う。初期設定は次とする。

```yaml
llm:
  base_url: http://ws1.local:11434/v1
  model: qwen3.6:35b-a3b-mtp-q4_K_M
  thinking: true
  stream: true
  request_timeout_seconds: 3600
  first_event_timeout_seconds: 600
  idle_timeout_seconds: 900
```

`base_url`と`model`は設定または環境変数で変更できる。OpenAIクライアントはAPIキーを必須とするが、Ollamaでは値を使わないため`ollama`を渡す。

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://ws1.local:11434/v1",
    api_key="ollama",
    timeout=3600,
)
```

### Thinkingとストリーミング

すべて `POST /v1/chat/completions` を使う。Thinkingとストリーミングは常に有効にする。

```python
stream = client.chat.completions.create(
    model=settings.llm.model,
    messages=messages,
    response_format={"type": "json_object"},
    stream=True,
    extra_body={"think": True},
)
```

実測では、`qwen3.6:35b-a3b-mtp-q4_K_M`のOpenAI互換ストリームは、本文より先に`delta.reasoning`へthinkingの断片を返した。受信処理はすべてのchunkから次を集める。

- `delta.reasoning`: thinkingの断片
- `delta.content`: 最終的な本文またはJSONの断片
- 終了情報

thinkingは本文として使わない。本文は`delta.content`だけを連結して読む。

### 無応答の判定

本文がまだ届いていなくても、thinkingが届いている間は正常に実行中と扱う。

- 接続から最初のthinkingまたは本文まで: 10分待つ
- 最初の受信後、thinking・本文・終了情報のいずれも届かない時間: 15分で停止する
- 呼び出し全体の上限: 1時間

受信したchunkごとに、時刻・種別・文字数だけを記録し、thinkingまたは本文を受け取るたびに最終受信時刻を更新する。thinking本文は保存・表示しない。画面やコマンドラインには「思考中」「本文生成中」「保存中」の進捗だけを出す。

### JSONの検証

構造化した応答が必要な依頼では、`response_format`だけに頼らない。依頼文の末尾にも、その呼び出し専用の小さなJSON Schemaを含める。

本文のstreamが終了した後、次の順に扱う。

1. `delta.content`を連結する。
2. JSONとして厳密に読み取る。
3. アプリ側のモデル定義で必須項目・型・文字列長を検証する。
4. `result`、`data`、`series`、`volume`、`scene`の入れ子に対象のJSONがあり、検証に通る場合だけ取り出す。
5. 読み取りまたは検証に失敗した場合は、同じ依頼を一度だけ再試行する。
6. 二度目も失敗した場合は、その作業単位で停止し、保存済みの状態を残す。

JSON Schemaは浅く保つ。深い入れ子、大量の選択肢、複雑な正規表現は使わない。

### LLMへの依頼単位

| 順番 | 依頼 | 応答 | 目的 |
|---:|---|---|---|
| 1 | 全巻計画 | 巻数、巻の役割、章・場面の配分、結末 | シリーズの骨格を決める |
| 2 | 巻の詳細計画 | 章、場面カード、巻内の変化 | 一巻を執筆可能な単位へ分ける |
| 3 | 場面執筆 | 本文、引継ぎ要約 | 物語本文を作る |
| 4 | 巻の要約 | 人物関係、未解決の問い、重要な事実 | 次巻へ必要な情報を縮める |
| 5 | 必要時の場面修正 | 修正後の本文、引継ぎ要約 | 壊れた形式または空本文を直す |

場面執筆の応答は次の小さなJSONに限定する。

```json
{
  "content": "場面本文",
  "handoff_summary": "次の場面に渡す要約"
}
```

## 5. 生成失敗時

本文が空、必要な見出しが欠ける、JSONが読めない、または場面の必要項目が欠ける場合は、その場面だけ再試行する。

再試行は1回までとする。二度目も失敗した場合は、シリーズ全体を破棄せず、その場面で停止する。

表現の好み、軽い矛盾、細かな文章品質は停止条件にしない。必要なら一度だけ書き直し、それでも残る問題は記録して先へ進む。

## 6. 保存と再開

実装では、次だけを保存する。

- 初回企画
- 全巻・章・場面の計画
- 各場面の本文と引継ぎ要約
- 各依頼の生の入力と最終本文応答
- ストリーミングの受信時刻・種別・文字数
- 完了した最後の場面
- 停止理由。停止した場合だけ

同じ設定を複数形式で複写しない。再開時は、最後に完了した場面の次から始める。

## 7. 出力

出力先の初期形は次とする。

```text
output/
  volume-01.md
  volume-02.md
  ...
  volume-10.md
  series.md
```

実際に作る巻数に対応するファイルだけを出力する。

各巻には巻題、章題、本文を含める。`series.md`には全巻を順番に収録する。販売用情報の自動作成、EPUB生成、KDPへの入稿は行わない。

## 8. KDP向けの扱い

初巻を無料導入巻として扱うが、KDPでの無料配布の実現方法は、公開時点のKDP規約と販促制度に従う。

初期版はKDPへの自動入稿を行わない。出力したMarkdownを利用者が確認・整形して公開する。
