# Storycraft 製品仕様

> この文書は、次の正式リリースで Storycraft が提供する振る舞いを定める。
> 実装言語、ライブラリ、モデル、通信方式、ファイル内部構造は定めない。
> 変えてはいけない製品上の約束は[要件](REQUIREMENTS.md)を参照する。

## 1. 範囲

Storycraft は、利用者の手入力briefまたは自由keywordsからLLMが生成するbriefを起点に、日本語の完結小説シリーズを自動生成し、各巻と全巻の Markdown 原稿を出力する。

- 利用者は開始時に手入力briefまたはkeywordsのどちらか一方を一度だけ渡す。
- keywordsは内容の種類を制限しない。生成briefは創作内容を検証・採点せず、手入力briefと同じ構造契約だけを通す。
- 通常実行中に、巻・章・場面ごとの確認は求めない。
- 中断した実行は、同じ企画を再入力せずに再開できる。
- KDP などへの自動入稿・自動公開は行わない。

## 2. 入力

初回入力は次のどちらか一方とする。

1. 人が作成したbrief YAML/JSON。JSON は YAML として読める形式であれば受け付ける。
2. 任意個数の自由なkeywords。ジャンル、巻数、雰囲気、題材、人物像、避けたい要素などを混在できる。LLMが同じbrief構造を生成する。

```yaml
title: 仮題
genre: ジャンルと雰囲気
protagonist: 主人公
key_people: 重要人物
want: 起きてほしいこと
avoid: 避けたいこと
ending: 最後にどう終えたいか
volumes: 5 # 省略可
```

必須項目は `title`、`genre`、`protagonist`、`key_people`、`want`、`avoid`、`ending` とする。

`volumes` の扱いは次のとおりとする。

| 入力 | 扱い |
|---|---|
| `4`〜`10` の整数 | 指定巻数として固定する。計画と出力の巻数は必ず一致する。 |
| 省略 | 物語に合わせて 4〜10 巻から選ぶ。5 巻を推奨値とする。 |
| それ以外 | 実行開始前に入力エラーとして停止する。 |

`chapters_per_volume` は省略可能な整数配列である。指定時は各要素を巻番号に対応する章数として扱い、対象巻の章一覧がこの配列と完全に一致しなければならない。配列長は指定巻数と一致し、各要素は `1`〜`12` とする。省略時は、各巻の章一覧工程が章数を決める。

## 3. 生成の流れ

シリーズは、次の順に作る。全巻本文、全章の細部、全場面を一度に作ってはならない。

0. keywords時だけ、LLMによるbrief生成と手入力briefと同じ構造契約の検証
1. 人物の初期台帳
2. 人物関係の初期台帳
3. 世界・場所・重要物の初期台帳
4. 時間・期限の初期台帳
5. 主要な伏線・問い・重要イベントの初期台帳
6. 初期台帳の横断整合確認とCanon確定
7. Canonの既存thread IDを巻別に配分するvolume_map
8. 巻ごとの章一覧
9. 章ごとの場面カード
10. 場面本文
11. 本文根拠つき継続性要約・状態更新の抽出
12. 巻の要約
13. 完結確認
14. Markdown 出力

`volume_map` は、確定Canonにあるthread IDだけを各巻の `introduce`、`advance`、`resolve` 操作へ配分する。新しい人物、設定、因果、秘密、出来事、結末条件を作らない。major項目は導入から回収までの配分を持つ。最終巻以外の表示用 `reader_question` は空にせず、最終巻は空文字列とする。結末到達条件の唯一の正本はbriefの `ending` であり、volume_mapはこれを複写・要約しない。

初期台帳は本文より前に確定するシリーズの正本である。人物、関係、世界、時間、主要項目の固定情報と開始時点の現在状態を持つ。初期台帳を確定した後、後続工程は固定情報、秘密の真実、回収条件を変更できない。

章一覧は各巻の開始時に、その巻の役割、確定済み台帳、前巻までの要約を入力として作る。全巻分の詳細章一覧を初回に作らない。

各LLM工程は、生成、検証、批評、必要時の修正、再検証、採用または検証済み草稿保持、保存の順に扱う。

- 生成は、工程に必要な情報だけを入力にする。
- 検証に失敗した生成結果は採用しない。
- 批評は修正可能で根拠のある issue だけを出す。称賛、点数、出版可否は出さない。
- 修正版は検証を通り、草稿の必須情報を失わない場合だけ採用する。そうでなければ草稿を保持する。
- 改善の試行回数は設定できる。批評または修正の失敗は検証済み草稿を破棄する理由にしない。

## 4. 状態と再開

実行中の状態は、途中で停止しても次の単位から続行できるように保存する。

保存するものは次である。

- 初回企画
- 全巻計画、章一覧、場面カード
- 各台帳の現在状態
- 各場面の本文、引継ぎ要約、状態更新
- 巻の要約
- 完了した工程と最後に完了した場面
- 停止理由
- 生成・検証・批評・修正に使った入出力と検証結果、および対応するraw記録への参照

作業場所は一つのシリーズだけを保持する。`resume` と `step` は、指定した作業場所の保存済み状態だけを対象にする。

同じ作業場所の `run`、`resume`、`step` は同時に実行できない。別の実行が使用中なら、状態や出力を変更せずに拒否する。

状態は、一時ファイルへの書込みと同期を終えてから原子的に置き換える。再開時は、保存済みの正本状態から未完了の実行単位を続ける。完了済みの成果物を作り直してはならない。

## 5. 台帳

台帳は、本文のための補助メモではなく、シリーズの継続性を保つ正本である。

| 台帳 | 管理するもの |
|---|---|
| 人物 | 固定プロフィール、現在の目的・状態・知識 |
| 関係 | 人物間の現在の関係と読者が知る関係 |
| 世界 | 場所、組織、重要物、世界の規則と現在状態 |
| 時間 | 時点、期限、移動、回復、同時進行イベント |
| 伏線 | 謎、秘密、約束、関係、脅威、手掛かりと回収条件 |

すべての台帳項目は変更しない ID を持つ。ID はプログラムが割り当て、削除後も再利用しない。LLM は新規項目の内容を返してよいが、ID を決めてはならない。

台帳には、次の二種類の情報を分けて持つ。

- **固定情報**: 名前、物語上の役割、確定事実、秘密の真実、回収条件など。場面執筆は変更できない。
- **現在状態**: 現在地、目的、負傷、関係、期限の進行、伏線の導入・進展・回収など。許可された場面更新でだけ変更できる。

## 6. 場面の契約

場面は、本文、引継ぎ要約、状態更新を一つの採用成果物として扱う。本文工程は完成本文だけを返し、その後の継続性抽出工程が凍結本文を根拠に要約と更新候補を返す。三者の検証がそろうまで、本文だけ、要約だけ、状態更新だけを独立して採用してはならない。

各場面カードは、少なくとも次を定める。

- 視点人物
- 開始時刻と終了時刻
- 場所
- 場面の目的と必須イベント
- 登場人物
- 導入・進展・回収する伏線
- 読者に見せてよい情報
- 見せ方と、見せてはいけない情報
- 場面終了時に変わること
- シリーズ内で一意な場面ID

場面生成、批評、修正へ渡せる情報は、場面カードで参照された項目と、それらを理解するために直接必要な情報だけとする。関係のない人物、伏線、秘密、将来の状態を渡してはならない。

場面の状態更新には次の規則を適用する。

- 更新できるのは、場面カードで対象になった人物、関係、世界、時間、伏線だけである。
- 更新できるのは、その台帳で現在状態として定義された項目だけである。
- 固定情報、秘密の真実、回収条件は更新できない。
- 更新には、場面カードの場面IDを記録する。
- 場面を採用するときは、状態更新を検証して正本台帳へ適用し、次の場面は更新済み台帳を参照する。

## 7. 知識と情報開示

伏線や秘密を扱うため、少なくとも次を区別する。

1. 作者だけが知る真実
2. 視点人物が現在知っていること
3. 読者が現在知っていること
4. この場面で新しく見せてよいこと

視点人物が知らない事実を、その人物の内心や断定的な地の文で書いてはならない。秘密を知る人物が視点人物でも、まだ明かしてはいけない真実は、行動、沈黙、会話、観察などで表す。

伏線には、導入期限、回収期限、回収条件、現在の状態を持たせる。主要な伏線は、導入先と回収先が決まらなければ採用しない。

全巻計画で主要と明示した初回企画由来の問い、伏線、重要イベントだけを、主要項目として台帳へ登録する。完結確認は登録済みの主要項目を対象にする。

## 8. 検証と停止

次は硬い検証対象である。

- 入力の必須項目と範囲
- 構造、必須項目、型
- 既知 ID と許可された ID の参照
- 定義済みの状態値と操作値
- 場面カードと状態更新の整合
- 空本文、必要な章・場面の欠落、同一本文の重複
- 主要な伏線の未回収

硬い検証に失敗した生成結果は、同じ入力条件で再試行できる。規定回数に達しても成功しない場合は、その工程で停止し、保存済みの状態を残す。

次は品質指標であり、それだけでは停止・棄却しない。

- 本文量
- 表現の好み
- 軽い不整合
- 文章の冗長さ

品質上の問題は批評へ渡し、改善後も残る場合は記録して先へ進む。

## 9. 完結確認

最終出力の前に、初回企画由来の主要な問い、主要な伏線・重要イベント、結末を確認する。

- 主要項目ごとに、回収済みか未回収かを判定する。
- 未回収の主要項目があれば、シリーズを完成扱いにしない。
- 最終巻は、初回企画で指定した結末を回収して完結する。

## 10. 出力

出力先には、実際に作成した巻数に対応する Markdown と、全巻をまとめた Markdown を作る。

```text
output/
  volume-01.md
  volume-02.md
  ...
  series.md
```

各巻の Markdown には巻題、章題、本文を含める。`series.md` には全巻を順番に収録する。

第1巻は無料導入巻、第2巻以降は販売対象巻として識別する。入稿・公開そのものは行わない。

出力前に、空本文、必要な章・場面の欠落、同一本文の重複、未回収の主要項目、結末根拠の不足がないことを確認する。検証は一時出力に対して行い、成功時だけ既存の出力一式を原子的に置き換える。失敗時は既存の完成出力を変更しない。

## 11. 操作

利用者向けの操作は次の三つとする。

| 操作 | 役割 |
|---|---|
| `run` | 未保存の作業場所で、企画から最終出力までを連続実行する。 |
| `resume` | 指定作業場所の保存済み状態から、未完了の実行単位を連続実行する。 |
| `step` | 次の未完了の保存可能な単位を一つだけ実行する。初回だけ企画を指定する。完了済みなら出力を再生成しない。 |

## 12. 非目標

次はこの仕様の範囲に含めない。

- KDP を含む販売先への自動入稿・自動公開
- EPUB など Markdown 以外の販売形式の自動生成
- Web UI
- 複数シリーズの並列生成
- 売上や販売成果の保証
