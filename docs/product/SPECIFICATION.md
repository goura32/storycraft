# Storycraft 仕様

> この文書は、現在の実装方針を記載する。要件を守る範囲で変更してよい。
> 変えてはいけない製品上の約束は[要件](REQUIREMENTS.md)を参照する。

## 1. 初回入力

利用者は最初に次を入力する。

- 仮題
- ジャンルと雰囲気
- 主人公
- 重要人物
- 起きてほしいこと
- 避けたいこと
- 最後にどう終えたいか
- 巻数。省略時は5巻

巻数は4〜10の範囲だけを受け付ける。

## 2. 分量と構成

本文は一場面あたり約1,800〜2,600字を目安にする。章は原則3場面で構成する。

| 巻数 | 1巻の章数 | 1巻の場面数 | 1巻の本文目安 |
|---:|---:|---:|---:|
| 4巻 | 9章 | 27場面 | 49,000〜70,000字 |
| 5〜6巻 | 8章 | 24場面 | 43,000〜62,000字 |
| 7〜8巻 | 7章 | 21場面 | 38,000〜55,000字 |
| 9〜10巻 | 6章 | 18場面 | 32,000〜47,000字 |

章の途中で物語上の区切りが必要な場合だけ、1章を2または4場面にしてよい。各巻は少なくとも6章と18場面を持つ。

## 3. 自動実行

1. 全巻の計画を作る。
2. 各巻の目的、人物関係の変化、巻末の引きを決める。
3. 各巻を章と場面へ分ける。
4. 場面を順番に書き、本文と次の場面への短い引継ぎ要約を保存する。
5. 巻末で本文をまとめる。
6. 最終巻の後に全巻をまとめる。

通常の実行では、途中確認を求めない。利用者が明示的に停止した場合だけ中断する。

## 4. Ollamaの利用方法

### 接続設定

OllamaのOpenAI互換APIを使う。初期設定は次とする。

```yaml
llm:
  base_url: http://ws1.local:11434/v1
  model: qwen3.6:35b-a3b-mtp-q4_K_M
  timeout_seconds: 3600
```

`base_url`と`model`は設定または環境変数で変更できる。OpenAIクライアントはAPIキーを必須とするが、Ollamaでは値を使わないため`ollama`を渡す。

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://ws1.local:11434/v1",
    api_key="ollama",
    timeout=3600,
)
```

本文を含む生成は時間がかかるため、呼び出しごとの制限時間は1時間とする。途中で止まっても、完了済みの場面から再開する。

### 呼び出すAPI

すべて `POST /v1/chat/completions` を使う。ストリーミングは使わない。

```python
completion = client.chat.completions.create(
    model=settings.llm.model,
    messages=messages,
    response_format={"type": "json_object"},
    stream=False,
)
```

構造化した応答が必要な依頼では、`response_format`だけに頼らない。依頼文の末尾にも、その呼び出し専用の小さなJSON Schemaを含める。

応答は次の順に扱う。

1. 生の依頼と応答を保存する。
2. JSONとして厳密に読み取る。
3. アプリ側のモデル定義で必須項目・型・文字列長を検証する。
4. `result`、`data`、`series`、`volume`、`scene`の入れ子に対象のJSONがある場合だけ、検証に通るものを取り出す。
5. 読み取りまたは検証に失敗した場合は、同じ依頼を一度だけ再試行する。
6. 二度目も失敗した場合は、その作業単位を停止し、保存済みの状態を残す。

JSON Schemaは浅く保つ。深い入れ子、大量の選択肢、複雑な正規表現は使わない。

### LLMへの依頼単位

| 順番 | 依頼 | 応答 | 目的 |
|---:|---|---|---|
| 1 | 全巻計画 | 巻題、巻の目的、関係変化、巻末の引き | シリーズの骨格を決める |
| 2 | 巻ごとの章計画 | 章題、章の役割、場面の目的 | 一巻を分割する |
| 3 | 場面執筆 | 本文、引継ぎ要約 | 物語本文を作る |
| 4 | 必要時の場面修正 | 修正後の本文、引継ぎ要約 | 壊れた形式または空本文を直す |

LLMに一度に全巻本文や全章の細部を作らせない。本文生成時は、次だけを渡す。

- その場面の目的
- 必須の出来事
- 登場人物と関係の抜粋
- 直前場面の引継ぎ要約
- 最終場面だけ、最初に決めた結末

LLMから受け取る本文は、次の小さなJSONに限定する。

```json
{
  "content": "場面本文",
  "handoff_summary": "次の場面に渡す要約"
}
```

## 5. 生成失敗時

本文が空、必要な見出しが欠ける、JSONが読めない、または場面の必要項目が欠ける場合は、その場面だけ再試行する。

再試行は1回までとする。二度目も失敗した場合は、シリーズ全体を破棄せず、その場面で停止する。

表現の好み、軽い矛盾、細かな文章品質は停止条件にしない。必要なら一度だけ書き直し、それでも残る問題は記録して先へ進む。

## 6. 保存と再開

実装では、次だけを保存する。

- 初回企画
- 全巻・章・場面の計画
- 各場面の本文と引継ぎ要約
- 各依頼の生の入力と応答
- 完了した最後の場面
- 停止理由。停止した場合だけ

同じ設定を複数形式で複写しない。再開時は、最後に完了した場面の次から始める。

## 7. 出力

出力先の初期形は次とする。

```text
output/
  volume-01.md
  volume-02.md
  ...
  volume-10.md
  series.md
```

実際に作る巻数に対応するファイルだけを出力する。

各巻には巻題、章題、本文を含める。`series.md`には全巻を順番に収録する。販売用情報の自動作成、EPUB生成、KDPへの入稿は行わない。

## 8. KDP向けの扱い

初巻を無料導入巻として扱うが、KDPでの無料配布の実現方法は、公開時点のKDP規約と販促制度に従う。

初期版はKDPへの自動入稿を行わない。出力したMarkdownを利用者が確認・整形して公開する。
